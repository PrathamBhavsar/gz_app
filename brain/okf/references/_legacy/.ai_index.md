## Tech Stack
| Concern | Package | Notes |
|---|---|---|
| State | `flutter_riverpod` | Plain `Provider`, `Notifier`, and `AsyncNotifier`. No Riverpod codegen. |
| HTTP | `http` via `lib/core/api/api_client.dart` | API access stays below the notifier layer. |
| Models | Manual classes in `lib/models/` | Handwritten `fromJson`; no `freezed` or `json_serializable`. |
| Routing | `go_router` | Routes and path builders live in `lib/core/navigation/routes.dart`. |
| UI/Sizing | Custom responsive primitives in `lib/core/responsive/` | No `flutter_screenutil`; use theme and spacing tokens directly. |

## Architecture Contract
1. Feature data flow is `Widget -> Notifier -> Repository -> ApiClient -> API`.
2. `lib/models/` is the canonical shared model layer for both player and admin.
3. Shared UI states are mandatory for data-bound screens: loading, error, empty, and data.
4. `brain/code_map.md` and the relevant registry file must be updated in the same change when `.dart` files are added, moved, or deleted.

## Current Reality

  `application/` notifiers wired to the live backend (Player + Admin). No real dummy data remains
